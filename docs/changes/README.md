# Canbox 变更记录系统

## 📋 概述

本文档记录 Canbox 项目的所有功能变更、技术改进和设计决策。变更记录分为三个状态：

1. **进行中** (`active/`) - 正在实施或讨论中的变更
2. **已完成** (`completed/`) - 已成功实施并发布的变更
3. **归档** (`archive/`) - 历史变更记录，按年份归档

## 📁 目录结构

```
changes/
├── README.md              # 本文件，变更系统说明
├── template.md           # 变更记录模板
├── active/               # 进行中的变更
│   ├── 2026-04-operation-feedback.md
│   └── ...其他进行中变更
├── completed/            # 已完成的变更
│   ├── README.md         # 已完成变更汇总
│   ├── 2026-04-operation-feedback.md
│   ├── 2026-03-log-viewer.md
│   └── ...其他已完成变更
└── archive/              # 归档的变更（历史记录）
    ├── 2025/            # 2025年变更
    └── 2026/            # 2026年变更
```

## 📝 变更记录格式

每个变更记录应包含以下部分：

### 变更标题和基本信息

```markdown
# [变更标题] - [YYYY-MM-DD]

## 📋 基本信息
- **状态**: ✅ 已完成 / 🚧 进行中 / 📋 计划中
- **优先级**: ⭐⭐⭐⭐⭐ (1-5星)
- **难度**: 🟢 简单 / 🟡 中等 / 🔴 困难
- **预计时间**: X小时/天
- **实际时间**: X小时/天
- **负责人**: [姓名/团队]
```

### 变更内容

```markdown
## 🎯 变更概述
[简要描述变更的目标和价值]

## 🔍 问题描述
[详细说明要解决的问题或改进点]

## 💡 解决方案
[描述实现方案和技术选择]

## 📁 修改的文件
[列出所有修改的文件，按模块分类]

## 🧪 测试验证
[描述测试方法和验证结果]

## 📊 影响评估
[评估变更对系统的影响]

## 🔗 相关链接
[链接到相关文档、Issue、PR等]
```

## 🔄 变更流程

### 1. 创建变更记录

当开始一个新功能或改进时：

1. 复制 `template.md` 创建新文件
2. 填写变更基本信息
3. 保存到 `active/` 目录
4. 文件名格式：`YYYY-MM-变更名称.md`

### 2. 实施变更

在实施过程中：

1. 更新变更记录的实施细节
2. 记录遇到的问题和解决方案
3. 跟踪实际花费时间

### 3. 完成变更

变更实施完成后：

1. 更新状态为"已完成"
2. 填写实际时间和验证结果
3. 将文件移动到 `completed/` 目录
4. 更新 `completed/README.md` 汇总

### 4. 定期归档

每季度或每年：

1. 将旧的已完成变更移动到 `archive/` 目录
2. 按年份组织归档文件
3. 更新归档索引

## 🔗 与 OpenSpec 集成

本变更记录系统与 OpenSpec 系统协同工作：

1. **OpenSpec** - AI友好的详细变更记录

   - `openspec/changes/[change-id]/`
   - 包含：proposal.md, design.md, tasks.md, README.md
   - 用于AI Agent处理和自动化工作流
2. **本系统** - 人类友好的简化记录

   - `docs/changes/[status]/[change].md`
   - 提取OpenSpec关键信息生成简化文档
   - 便于人类开发者快速了解变更内容

### 同步机制

每次OpenSpec变更完成后，AI Agent自动：

1. 读取OpenSpec变更内容
2. 提取关键信息
3. 生成简化Markdown文档
4. 保存到本系统相应目录
5. 更新汇总文档

## 📈 质量指标

### 变更质量评估

每个变更完成后应评估：

1. **代码质量** - 符合编码规范，无技术债务
2. **测试覆盖** - 有充分的测试用例
3. **文档完整** - 相关文档已更新
4. **性能影响** - 对系统性能无负面影响
5. **向后兼容** - 不影响现有功能

### 成功率统计

- **一次成功率**: 变更首次实施即成功的比例
- **返工率**: 需要修改或重新实施的变更比例
- **平均实施时间**: 各类变更的平均实施时间

## 🚀 快速开始

### 查看变更记录

```bash
# 查看所有已完成变更
cat docs/changes/completed/README.md

# 查看特定变更详情
cat docs/changes/completed/2026-04-operation-feedback.md

# 查看进行中变更
ls docs/changes/active/
```

### 创建新变更

1. 复制模板：

```bash
cp docs/changes/template.md docs/changes/active/2026-04-新功能名称.md
```

2. 编辑变更内容
3. 开始实施

### 更新变更状态

变更完成后：

```bash
mv docs/changes/active/2026-04-新功能名称.md docs/changes/completed/
```

## 🆘 支持与帮助

### 常见问题

**Q: 变更记录应该多详细？**
A: 应包含足够信息让其他开发者理解变更内容，但不必像OpenSpec那样详细。

**Q: 如何处理小的Bug修复？**
A: 小的Bug修复可以合并记录，如"2026-04-多项Bug修复.md"。

**Q: 变更记录需要谁审批？**
A: 重要变更需要技术负责人或项目经理审批。

**Q: 如何关联变更和代码提交？**
A: 在变更记录中添加相关的PR链接和Commit Hash。
