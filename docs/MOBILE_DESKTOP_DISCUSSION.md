# Canbox 移动端与桌面端发展讨论

## 1. Electron 的移动限制

### 现状
- Electron 基于 Chromium + Node.js，仅支持桌面平台（Windows/macOS/Linux）
- 无官方移动支持计划，社区方案（如 `electron-mobile`）已停止维护

### 核心原因
| 限制项 | 说明 |
|--------|------|
| 资源占用 | Electron 包体积大（>100MB），内存占用高，不适合移动设备 |
| 原生能力 | 移动端需要相机、GPS、传感器等，Electron 缺乏统一桥接方案 |
| 应用商店 | iOS 禁止动态代码执行（V8/JIT），导致 Electron 技术栈无法上架 |
| 用户体验 | 桌面交互模式（右键、悬浮窗、多窗口）与移动端差异巨大 |

---

## 2. 移动端技术栈选择

如果 Canbox 要进入移动端，需要考虑以下方案：

### 方案 A：独立原生 App
- **技术栈**：Swift/Kotlin 分别开发 iOS/Android
- **优点**：性能最佳，原生体验，应用商店合规
- **缺点**：开发成本高，需维护两套代码，与桌面端无法复用逻辑

### 方案 B：跨平台框架
| 框架 | 技术栈 | 与 Canbox 关系 | 可行性 |
|------|--------|----------------|--------|
| React Native | JavaScript/React | 可复用部分 JS 逻辑 | 高，但需重写 UI |
| Flutter | Dart | 独立生态，无法复用 | 中，学习成本高 |
| Taro/UniApp | Vue/React 转小程序/App | 可复用部分 Vue 组件 | 中，适合国内场景 |
| Capacitor/Cordova | WebView 壳 | 可包装现有 Web 代码 | 较高，但体验受限 |

### 方案 C：小程序生态
- 基于微信/支付宝小程序，使用 Canbox 作为"开发工具"
- 用户最终在小程序中使用应用，Canbox 仅用于开发调试
- **优点**：无需用户安装，分发成本低
- **缺点**：受平台限制，无法脱离微信生态

## 3. 两端打通的可能性分析

### 3.1 架构层面打通

```
┌─────────────────────────────────────────────────────────────┐
│                     Canbox 应用运行时                         │
├──────────────────┬──────────────────┬───────────────────────┤
│   桌面端 (Electron) │   移动端 (RN/Taro)  │    Web 端 (PWA)        │
│   - Chromium 渲染  │   - 原生组件渲染    │   - 浏览器渲染          │
│   - Node.js API   │   - Bridge 桥接    │   - 受限 API           │
└──────────────────┴──────────────────┴───────────────────────┘
│                     统一 JavaScript 业务逻辑                  │
│              (应用代码通过条件编译/运行时判断适配)              │
└─────────────────────────────────────────────────────────────┘
```

**可行路径**：
1. **业务逻辑层复用**：将 App 的核心逻辑抽离为纯 JS 库，不依赖 DOM/Node API
2. **UI 层分别实现**：桌面端用 Vue + Element，移动端用 Taro/Vant 等
3. **统一打包规范**：Canbox 应用格式（`manifest.json` + 代码包）在各端保持一致

### 3.2 实际挑战

| 挑战 | 严重程度 | 可能的解决方向 |
|------|----------|----------------|
| UI 交互差异 | 高 | 设计移动端专属组件库，不追求与桌面端一致 |
| 存储/文件系统 | 中 | 定义抽象存储接口，各端提供不同实现 |
| 窗口管理 | 高 | 移动端改为"页面栈"模型，放弃多窗口 |
| 系统 API 权限 | 中 | 移动端需申请相机/相册等权限，与桌面端不同 |
| 应用分发 | 高 | iOS 必须走 App Store，无法像桌面端自由安装 |

### 3.3 阶段性建议

**第一阶段：桌面端深耕**
- 完善桌面端体验，建立稳定的应用生态
- 抽象可复用的业务逻辑层

**第二阶段：Web 版试水**
- 提供 PWA 版本，验证移动端需求
- 受限但无需安装，可作为过渡方案

**第三阶段：移动端评估**
- 根据用户规模决定投入原生/跨平台开发
- 优先考虑 Taro/UniApp 方案，可复用 Vue 技术栈

---

## 4. 总结

1. **移动端**：短期内（1-2 年）不建议投入，Electron 无法移植，跨平台成本较高。可先通过 PWA 验证需求。

2. **长期愿景**：Canbox 可定位为"桌面端应用平台"，移动场景通过"导出为 PWA/小程序"间接支持，而非直接竞争移动端原生应用市场。
