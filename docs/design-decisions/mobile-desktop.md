# Canbox 移动端与桌面端发展讨论

## 1. Electron 的移动限制

### 现状
- Electron 基于 Chromium + Node.js，仅支持桌面平台（Windows/macOS/Linux）
- 无官方移动支持计划，社区方案（如 `electron-mobile`）已停止维护

### 核心原因

| 限制项 | 说明 |
|--------|------|
| 资源占用 | Electron 包体积大（>100MB），内存占用高，不适合移动设备 |
| 原生能力 | 移动端需要相机、GPS、传感器等，Electron 缺乏统一桥接方案 |
| 应用商店 | iOS 禁止动态代码执行（V8/JIT），导致 Electron 技术栈无法上架 |
| 用户体验 | 桌面交互模式（右键、悬浮窗、多窗口）与移动端差异巨大 |

---

## 2. 移动端技术栈选择

如果 Canbox 要进入移动端，需要考虑以下方案：

### 方案 A：独立原生 App
- **技术栈**：Swift/Kotlin 分别开发 iOS/Android
- **优点**：性能最佳，原生体验，应用商店合规
- **缺点**：开发成本高，需维护两套代码，与桌面端无法复用逻辑

### 方案 B：跨平台框架

| 框架 | 技术栈 | 与 Canbox 关系 | 可行性 |
|------|--------|----------------|--------|
| **React Native** | JavaScript + React | 与 Canbox 前端技术栈相似，可复用部分代码 | 🟢 高 |
| **Flutter** | Dart | 需要学习新技术栈，与现有技术栈差异大 | 🟡 中 |
| **Ionic/Capacitor** | Web 技术 + 原生包装 | 可复用最多前端代码，但性能较差 | 🟡 中 |
| **Uni-app** | Vue.js | 与 Canbox 前端技术栈一致，但生态系统较小 | 🟡 中 |

### 方案 C：PWA（渐进式 Web 应用）
- **技术栈**：纯 Web 技术（HTML/CSS/JS）
- **优点**：开发成本最低，无需应用商店审核，跨平台一致
- **缺点**：功能受限（无法访问所有原生 API），离线能力有限
- **与 Canbox 关系**：可复用 90% 前端代码，但需要重新设计 API 层

## 3. 架构设计选择

### 统一架构 vs 独立架构

#### 统一架构（共享代码库）
```
┌─────────────────────────────────────────────────┐
│               共享代码库 (Shared Core)           │
│  - 业务逻辑                                      │
│  - 数据模型                                      │
│  - 工具函数                                      │
└─────────────────────────────────────────────────┘
            ├───────────────────────┤
┌─────────────────────┐    ┌─────────────────────┐
│   桌面端适配层      │    │   移动端适配层      │
│  - Electron 桥接    │    │  - React Native 桥接│
│  - 桌面特有 UI      │    │  - 移动特有 UI      │
└─────────────────────┘    └─────────────────────┘
```

**优点**：
- 代码复用率高
- 业务逻辑一致
- 维护成本较低

**缺点**：
- 架构复杂
- 需要抽象适配层
- 可能影响性能

#### 独立架构（分离代码库）
```
桌面端 (Electron)               移动端 (React Native)
┌─────────────────┐            ┌─────────────────┐
│ 独立代码库      │            │ 独立代码库      │
│ - 前端: Vue.js  │            │ - 前端: React   │
│ - 后端: Node.js │            │ - 桥接: Native  │
└─────────────────┘            └─────────────────┘
```

**优点**：
- 架构简单清晰
- 各平台优化自由度高
- 技术栈选择灵活

**缺点**：
- 代码重复
- 业务逻辑可能不一致
- 维护成本翻倍

## 4. 数据同步方案

### 4.1 云端同步
```
桌面端 ────▶ 云端服务器 ◀──── 移动端
   │           │               │
   │           │               │
   ▼           ▼               ▼
本地存储     数据库         本地存储
```

**技术方案**：
- **Firebase**：实时数据库，易于集成
- **Supabase**：开源替代，PostgreSQL 基础
- **自建服务**：最大控制权，但维护成本高

### 4.2 本地同步
```
桌面端 ─────── Wi-Fi/蓝牙 ─────── 移动端
   │                               │
   │                               │
   ▼                               ▼
本地存储                         本地存储
```

**技术方案**：
- **WebRTC**：P2P 通信，无需服务器
- **局域网发现**：mDNS/Bonjour
- **文件共享**：通过共享文件夹同步

### 4.3 混合方案
- 主要数据：云端同步（保证可用性）
- 临时数据：本地同步（保证速度）
- 敏感数据：端到端加密

## 5. 功能兼容性分析

### 5.1 完全兼容的功能
| 功能 | 桌面端 | 移动端 | 说明 |
|------|--------|--------|------|
| 应用管理 | ✅ | ✅ | 列表、安装、卸载 |
| 设置管理 | ✅ | ✅ | 主题、语言等 |
| 数据存储 | ✅ | ✅ | 使用统一存储 API |

### 5.2 部分兼容的功能
| 功能 | 桌面端 | 移动端 | 说明 |
|------|--------|--------|------|
| 文件操作 | ✅ | ⚠️ | 移动端文件系统访问受限 |
| 系统集成 | ✅ | ⚠️ | 移动端原生 API 不同 |
| 多窗口 | ✅ | ❌ | 移动端通常单窗口 |

### 5.3 不兼容的功能
| 功能 | 桌面端 | 移动端 | 说明 |
|------|--------|--------|------|
| 系统托盘 | ✅ | ❌ | 移动端无系统托盘 |
| 全局快捷键 | ✅ | ❌ | 移动端不支持 |
| 命令行工具 | ✅ | ❌ | 移动端无命令行 |

## 6. 实施路线图

### 阶段 1：架构设计与原型（1-2个月）
1. **技术选型**：确定移动端技术栈
2. **架构设计**：设计统一的代码架构
3. **原型开发**：开发最小可行产品

### 阶段 2：核心功能迁移（3-4个月）
1. **共享核心**：提取可复用的业务逻辑
2. **UI 适配**：开发移动端 UI 组件
3. **API 桥接**：实现移动端原生 API

### 阶段 3：功能完善与优化（2-3个月）
1. **性能优化**：移动端性能调优
2. **功能完善**：补全缺失功能
3. **测试验证**：全面测试和用户反馈

### 阶段 4：发布与维护（持续）
1. **应用商店**：提交到 iOS/Android 商店
2. **用户反馈**：收集和分析用户反馈
3. **持续迭代**：根据反馈持续改进

## 9. 决策建议

### 推荐方案：渐进式混合架构
